## 2. Product Vision & Value Proposition
Lokality AI is an enterprise SaaS Recruitment Intelligence Platform that automates candidate interview screening using state-machine-driven AI interviewer personas, automated proctoring telemetry, and automated post-interview grading rubrics.

### Core Value Drivers:
1.  **Workflow-Driven Screening**: Moving away from CRUD operations to a strictly enforced candidate lifecycle state machine (applied ➔ parsed ➔ matched ➔ scheduled ➔ interviewed ➔ evaluated ➔ review).
2.  **Adaptive AI Conversations**: Combining structured, curated questions with dynamic AI-generated follow-ups under 40 words, establishing a conversational but standardized screening.
3.  **Proctoring & Integrity Enforcement**: Real-time webcam telemetry, tab-switch tracking, focus loss checks, and automated screenshot snapshots to prevent cheating without installing heavy client-side plugins.
4.  **Local-First / Cost-Isolated Development**: Out-of-the-box support for developer fallback mocks (Gemini mock, Resend log fallback, Supabase JWT decode) to build, test, and debug without relying on live database keys or active external API costs.

## 4. Functional Requirements

### 4.1. AI Resume Parsing & Vector Matching
*   **FR-1.1**: The system must extract candidate details (name, email, phone, experience years, skills list, resume summary) from uploaded PDFs and Word documents.
*   **FR-1.2**: If the Gemini API key is absent, the system must run a local heuristic text parser fallback extracting standard attributes and email matches.
*   **FR-1.3**: The system must calculate a candidate-to-JD match percentage score (0-100%) and output a contextual fit analysis based on skills overlap and requirements.

### 4.2. Interview Template & Persona Builder
*   **FR-2.1**: Recruiters must be able to configure AI Personas with customized system prompts (governing tone, language, and guidelines), difficulty levels, and duration.
*   **FR-2.2**: The platform must support "Interview Templates" that group JD ID, Persona config, curated questions lists, proctoring policies, and scoring criteria.
*   **FR-2.3**: Prompts must be loaded from a database-seeded `PromptLibrary` so administrators can tweak AI guidelines dynamically without redeploying code.

### 4.3. Pre-Interview Candidate Portal & System Check
*   **FR-3.1**: Candidates entering through their unique URL must complete a multi-step pre-flight check (Webcam check, microphone permissions check, browser focus lock check).
*   **FR-3.2**: Candidates must be prompted to grant focus permissions and receive clear warnings that changing browser tabs or leaving the screen will log integrity violations.

### 4.4. Live AI Interview Engine (State-Machine Driven)
*   **FR-4.1**: Interview sessions must follow a strict sequential state transition (`scheduled` ➔ `in_progress` ➔ `evaluation_pending` ➔ `completed` / `failed`).
*   **FR-4.2**: WebSocket gateway (`InterviewGateway`) must maintain the active session state, relaying transcript entries and proctor flags.
*   **FR-4.3**: Question Generation must prioritze scheduled curated questions first, transitioning into adaptive follow-ups generated by Gemini 2.5 Flash once curated questions are exhausted.
*   **FR-4.4**: AI questions must remain concise (under 40 words) and follow the persona's tone config.

### 4.5. Proctoring & Session Integrity Tracking
*   **FR-5.1**: The client-side dashboard must capture webcam screenshot snapshots every 15 seconds during active sessions and stream them to Supabase Storage.
*   **FR-5.2**: The system must capture and log client events: `tab_switch`, `focus_loss`, `devtools_open`, and `multi_face_detected`.
*   **FR-5.3**: Recorded integrity events must be written onto the candidate's database timeline and integrated into the post-interview scoring calculations.

### 4.6. AI Evaluation & Post-Interview Grading
*   **FR-6.1**: Upon session closure, a background task must be queued (`finalize-evaluation`) to grade the candidate.
*   **FR-6.2**: The grading agent must compute:
    *   Overall Score (0-100)
    *   Recommendation (`strong_hire`, `hire`, `no_hire`, `strong_no_hire`)
    *   Strengths list and concerns list
    *   Competency Scores (rubric-scoped)
    *   Integrity Score (deducted automatically based on tab-switch counts and focus logs)
*   **FR-6.3**: The grading agent must produce a professional markdown-formatted assessment report detailing the candidate's fit.

## 5. Non-Functional Requirements

### 5.1. Security & Data Privacy
*   **NFR-1.1**: The platform must support multi-tenant isolation. No recruiter can access candidates, interviews, or reports belonging to another organization.
*   **NFR-1.2**: REST API routes must be protected via `SupabaseAuthGuard`, extracting claims, roles, and org scopes from JWT headers.
*   **NFR-1.3**: The system must log crucial changes (state changes, user updates, report releases) to the `audit_events` ledger for compliance tracking.

### 5.2. Performance & Scalability
*   **NFR-2.1**: Real-time communication must use WebSockets with automatic fallback pools.
*   **NFR-2.2**: Heavy long-running operations (such as AI resume parsing, video transcript evaluations, and report compiles) must be handled asynchronously via a background task manager.
*   **NFR-2.3**: During host development, the system must utilize an in-memory queue processor (`InMemoryQueueService`) to avoid external Redis/Docker dependencies, with interfaces allowing immediate swap-in of BullMQ/Redis for production deployment.

### 5.3. Development Fallbacks
*   **NFR-3.1**: The backend must boot and run tests successfully even if database connection strings are absent (using test mocking structures) or if third-party credentials (Gemini, Resend, JWT Secret) are unconfigured.

## 6. Out of Scope
*   **OOS-1**: Native video redirects to Microsoft Teams or Zoom standalone applications. All proctoring, webcam telemetry, and AI dialogue are hosted inside the Lokality AI interview room to maintain compliance and strict focus control. (Calendar metadata support is retained in templates for organization coordination).
*   **OOS-2**: Custom browser extension installations. Proctoring relies entirely on standard browser APIs (page visibility API, focus listeners, canvas captures).
